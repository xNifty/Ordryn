Overview

All endpoints live under {{basePath}}/api/v1/ and return JSON. Requests must include a valid API key (see Authentication). Date and time fields use your account timezone for display; due dates are stored and returned as YYYY-MM-DD.

Authentication

Send your API key in the Authorization header using the Bearer scheme: 

Authorization: Bearer YOUR_API_KEY

Keys are created on the profile page and shown in full only once at creation. Revoked keys stop working immediately. The API requires Redis for key validation and rate limiting.

Device authorization

Mobile and desktop clients can obtain an API key through a browser hand-off when you are already signed in on the web. The client requests a device code, opens the verification URL in a browser, and polls until you approve access.

Method Path Auth Description
POST/api/v1/auth/device/codePublic (API enabled + Redis)Start device authorization; returns device_code, user_code, and verification URLs
POST/api/v1/auth/device/tokenPublic (API enabled + Redis)Poll for approval; returns api_key once when approved
GET/auth/device?user_code=…Browser session (optional)Approve or deny the request in the browser

Start authorization

POST {{basePath}}/api/v1/auth/device/code
Content-Type: application/json

{"client_name": "Android app"}

Response includes verification_uri_complete (open in a browser), expires_in (600 seconds), and interval (poll every 5 seconds).

Poll for token

POST {{basePath}}/api/v1/auth/device/token
Content-Type: application/json

{
  "device_code": "DEVICE_CODE",
  "grant_type": "urn:ietf:params:oauth:grant-type:device_code"
}

While pending, the token endpoint returns 400 with {"error":"authorization_pending"}. On approval it returns api_key, name, and key_prefix once. Approving again with the same client_name rotates (revokes) the previous key for that name.

Errors

Failed requests return JSON with a stable error code and a human-readable message:

{
  "error": "unauthorized",
  "message": "Invalid or revoked API key."
}
HTTP error When
400invalid_requestBad parameters or JSON body
400authorization_pending / access_denied / expired_tokenDevice authorization polling states
401unauthorizedMissing, invalid, or revoked API key
403api_disabledREST API disabled in site settings
404not_foundResource not found (or not owned by you)
405method_not_allowedHTTP method not supported for this path
409name_conflict / limit_reachedSaved view name or per-user limit conflict
429rate_limit_exceededToo many requests (see Retry-After header)
503api_unavailableRedis unavailable
500internal_errorUnexpected server error

Rate limits

Limits apply per user (per API key owner) using a token bucket stored in Redis:

  • Read requests (GET): 120 requests per minute (refill ~2/sec)
  • Write requests (POST, PATCH, DELETE): 60 requests per minute (refill ~1/sec)

When limited, the response is 429 with a Retry-After header (seconds until you can retry).

Tasks

Task object

{
  "id": 42,
  "title": "Buy groceries",
  "description": "Milk, eggs, bread",
  "completed": false,
  "due_date": "2026-07-15",
  "project_id": 3,
  "project": "Personal",
  "priority": 2,
  "favorite": false,
  "position": 5,
  "tags": [
    { "id": 1, "name": "errands", "color": "#6c757d" }
  ],
  "created_at": "2026-07-01T14:30:00Z",
  "modified_at": "2026-07-10T09:00:00Z"
}

priority: 0 = None, 1 = Low, 2 = Medium, 3 = High. due_date is empty string when unset. project_id is omitted when the task has no project.

Response:

{
  "tasks": [ /* task objects */ ],
  "total": 128,
  "page": 1,
  "per_page": 50,
  "total_pages": 3
}

Create task

POST /api/v1/tasks

JSON body:

{
  "title": "New task",           // required
  "description": "",             // optional, max 1000 chars
  "due_date": "2026-07-20",      // optional, YYYY-MM-DD
  "project_id": 3,               // optional; omit or use 0 for no project
  "priority": 1,                 // optional, 0–3 (default 0)
  "completed": false,            // optional (default false)
  "favorite": false,             // optional (default false)
  "tag_ids": [1, 2]              // optional
}

Returns 201 Created with the new task object.

Get task

GET /api/v1/tasks/{id}

Returns a single task object.

Update task

PATCH /api/v1/tasks/{id}

Send only the fields you want to change. All fields are optional:

{
  "title": "Updated title",
  "description": "New description",
  "due_date": "2026-08-01",
  "clear_due_date": true,        // set true to remove due date
  "project_id": null,            // null or 0 clears project; number sets project
  "priority": 3,
  "completed": true,
  "favorite": true,
  "tag_ids": [1]                 // replaces all tags on the task
}

Returns the updated task object.

Delete task

DELETE /api/v1/tasks/{id}

Returns 204 No Content on success.

Reorder tasks

POST /api/v1/tasks/reorder

Updates manual sort order (position) within one favorite or non-favorite group, matching the web drag-and-drop behavior. Tasks cannot move across favorite groups. Listed IDs are rearranged among the slots they already occupy, so filtered lists (e.g. incomplete-only) do not overwrite unrelated tasks. 

{
  "task_ids": [12, 5, 9],   // required: new order for this page window
  "favorite": false,        // required: which group is being reordered
  "page": 1,                // optional; default 1
  "per_page": 50,           // optional; default 50, max 100
  "project": "3"            // optional: project id, or "none"/"0" for no project
}

Returns { "ok": true } on success. Relist tasks with GET /api/v1/tasks to read the new order.

Example: list incomplete tasks

curl -s -H "Authorization: Bearer YOUR_API_KEY" \
  "{{basePath}}/api/v1/tasks?status=incomplete&per_page=10"

Saved views

Saved views are named, reusable sets of task-list filters. They are private to the API key owner, and each user can store up to 20. Fetching a view returns its filters; apply it by passing those values as query parameters to GET /api/v1/tasks. Page numbers are not stored.

Saved view object

{
  "id": 12,
  "name": "Overdue work",
  "filter": {
    "project": "3",
    "status": "incomplete",
    "due": "overdue",
    "completed": "",
    "priority": "2",
    "tag": "7",
    "sort": "priority",
    "search": "release"
  },
  "sort_order": 0,
  "created_at": "2026-07-14T15:44:17Z",
  "updated_at": "2026-07-14T15:44:17Z"
}

id, created_at, and updated_at are server-managed. Names are trimmed, limited to 80 characters, and unique per user. sort_order must be between 0 and 2147483647.

Filter fields

FieldAccepted values
projectEmpty for all projects; a positive project ID; 0 or none for no project
statusEmpty, complete, completed, or incomplete
dueEmpty, overdue, today, week, or none
completedEmpty, or week for tasks completed in the last 7 days
priorityEmpty, or 0–3
tagEmpty, or a positive tag ID
sortEmpty for default ordering, or priority
searchAny string up to 500 characters

Filter values are trimmed. Named values are case-insensitive and normalized in responses; for example, completed in status is returned as complete.

List and get saved views

GET /api/v1/saved-views

Returns an array ordered by sort_order, then case-insensitively by name, then by id.

GET /api/v1/saved-views/{id}

Returns one saved view. Views owned by another user return 404 Not Found.

Create a saved view

POST /api/v1/saved-views

{
  "name": "Overdue work",
  "filter": {
    "status": "incomplete",
    "due": "overdue",
    "sort": "priority"
  },
  "sort_order": 0
}

name is required. filter and sort_order are optional. Without sort_order, the view is placed after the user's existing views. Returns 201 Created with the new object and a Location header.

Replace or update a saved view

PUT /api/v1/saved-views/{id}

Requires both name and filter. sort_order is optional and remains unchanged when omitted.

PATCH /api/v1/saved-views/{id}

Updates any supplied field and requires at least one of name, filter, or sort_order:

{
  "sort_order": 2
}

Delete a saved view

DELETE /api/v1/saved-views/{id}

Returns 204 No Content on success.

Saved view validation and conflicts

Request bodies must contain one JSON object, may be at most 1 MiB, and may not contain unknown fields. Duplicate names return 409 name_conflict; creating a 21st view returns 409 limit_reached.

Projects

GET /api/v1/projects

Returns a JSON array of project objects:

[
  { "id": 1, "name": "Work" },
  { "id": 2, "name": "Personal" }
]

Projects are managed through the web UI. Use project IDs when creating or updating tasks.

Tags

List tags

GET /api/v1/tags

Returns a JSON array of tag objects:

[
  { "id": 1, "name": "urgent", "color": "#dc3545" },
  { "id": 2, "name": "home", "color": "#198754" }
]

Create tag

POST /api/v1/tags

JSON body:

{
  "name": "errands"    // required, max 50 characters
}

Returns 201 Created with the tag object. Names are unique per user (case-insensitive); if a tag with the same name already exists, that tag is returned.

Delete tag

DELETE /api/v1/tags/{id}

Removes the tag and its associations on tasks. Returns 204 No Content on success.

Pass tag_ids when creating or updating tasks to assign tags.
